# Pop — Full Project Audit (2026-07-17)

Audited at commit `dd5aeef` (v0.3.8). Scope: all of `src/`, `tests/`, `.github/`, `scripts/`, `contracts/`, docs, and repo hygiene. Method: full manual read of every source file per area, cross-checked against `Pop.Core` contracts, plus verification of CI state on GitHub. The .NET SDK could not be installed in the audit environment (network policy), so no local build/test run; the latest `main` CI run (Windows, Linux, macOS incl. `swift test`) is green.

**Severity legend:** Critical = core feature broken or process-killing on a common path. High = crash/hang/data-loss on a realistic path, or a major functional gap. Medium = robustness, race, or integrity gap. Low = polish, hygiene, latent.

---

## Executive summary

The project is well-architected: `Pop.Core` is small, pure, and reasonably tested; layering matches `docs/ARCHITECTURE.md`; the settings contract is genuinely enforced by tests; the Windows update flow is correctly delegated to Velopack over HTTPS with no self-executed downloads; no injection or `pull_request_target`-class issues exist in CI; the macOS bridge marshaling/memory ownership is careful and correct (struct layouts verified field-by-field).

The problems cluster in three places:

1. **The macOS native shell has a critical coordinate-system inversion** — CG/AX top-left coordinates are run through a Cocoa bottom-left Y-flip — which breaks title-bar detection and snap placement. The gesture path has likely never worked end-to-end on real hardware. The mac updater also consumes the downloaded archive after URLSession has deleted it.
2. **Both the Windows and Linux hosts violate the project's own "fail soft on the hot path" principle**: unguarded loops/callbacks that can silently kill drag tracking (Linux) or freeze system-wide mouse input and get the hook removed by the OS (Windows), with no detection or recovery. Windows DPI awareness is declared in a way that has no effect on a WPF app, breaking mixed-DPI multi-monitor setups.
3. **The release pipeline's back half is weak**: nothing is code-signed/notarized, Linux packaging runs after the release is already public (and re-runs delete Linux assets), and `appimagetool` is downloaded unpinned and unverified into a job holding a write token to the branch that installed clients auto-update from.

Top 5 fixes by value: mac coordinate flip (one file + two call sites) · mac event-tap re-enable on timeout · Windows hook: move blocking work off the LL-hook callback and fix DPI manifest · Linux: guard the polling loop + install an X error handler · pin & checksum appimagetool and gate the release on Linux packaging.

## Critical

### C1. macOS: CG/AX coordinates double-flipped — gesture and placement broken
`src/Pop.App.Mac/NativeHost/Sources/PopMacApp/DesktopGeometry.swift:35-57`, used at `DragRuntimeController.swift:79` and `WindowSystem.swift:23,105-111,79-90`

`ScreenCoordinator.pointInTopLeftSpace`/`rectInTopLeftSpace` flip Y via `desktopMaxY - y`. That flip is correct for `NSScreen.frame`/`visibleFrame` (Cocoa, bottom-left origin) — but the same helpers are applied to `CGEvent.location` and AX `kAXPositionAttribute` rects, which are **already top-left-origin** Quartz coordinates. (The file even uses the raw CG location correctly at `WindowSystem.swift:26` for `AXUIElementCopyElementAtPosition`, then flips the same value for comparisons.) Consequences, single 1080p display:

- Title-bar hit test is inverted: a click on the real title bar is rejected ("Pointer was not over a title bar"); a click near the window's bottom passes.
- Snapped windows land at the wrong Y (tile rects derived from correctly-flipped NSScreen space are written back through a flip into AX top-left space — with a menu bar + Dock, ~55 px off, overlapping the Dock).
- Vertical velocity sign is inverted in samples fed to `SnapDecider`, so Option-throws project to the wrong monitor in stacked arrangements.
- Additionally, a CG↔Cocoa flip must pivot on the **primary screen's** height (`NSScreen.screens[0].frame.maxY`), not the max over all screens (`DesktopGeometry.swift:35-37`); with a display arranged above the primary, all conversions shift.

Fix is contained: make `DesktopGeometry` treat CG/AX inputs as already top-left, flip only NSScreen-derived rects, pivot on primary-screen height. This strongly suggests the mac gesture path needs a real-hardware smoke test before the next release.

## High

### H1. Windows: DPI awareness declared but never takes effect
`src/Pop.App.Windows/Pop.App.Windows.csproj:11`, `app.manifest`

`<ApplicationHighDpiMode>PerMonitorV2</ApplicationHighDpiMode>` only works through the WinForms `ApplicationConfiguration.Initialize()` bootstrap, which this app (custom `Main`, `App.xaml.cs:10`) never calls; `app.manifest` has no `<dpiAwareness>` element, so WPF's default System-DPI-aware wins. On mixed-DPI multi-monitor setups, `GetWindowRect`/`MoveWindow`/`GetMonitorInfo` return virtualized coordinates while `DwmGetWindowAttribute(DWMWA_EXTENDED_FRAME_BOUNDS)` (`Platform/Windowing/WindowSnapBoundsCalculator.cs:21-33`) returns physical ones — the inset math is wrong on any monitor whose scale differs from system DPI. `WindowInspector.IsCaptionHit` (`WindowInspector.cs:126-133`) also sends virtualized coords in `WM_NCHITTEST` to per-monitor-aware windows, so caption detection silently misfires on non-100% monitors. **Fix:** add `<dpiAwareness>PerMonitorV2</dpiAwareness>` to `app.manifest`.

### H2. Windows: blocking cross-process work inside the low-level mouse hook
`Platform/Input/MouseHookDragTracker.cs:40,125`, `PopHost.cs:256-259`, `WindowInspector.cs:126,216-247`

The LL hook runs on the dispatcher thread; every system-wide left-click triggers `SendMessageTimeout(WM_NCHITTEST, 50ms)` plus process/token queries inside the callback. Worse, `OnDragUpdated → TryRestorePreviousSnap` calls `_windowMover.MoveWindowAsync(...).GetAwaiter().GetResult()` inside the hook: `MoveWindow` on a foreign window synchronously dispatches to the target's thread — a hung app freezes system-wide mouse input until Windows exceeds `LowLevelHooksTimeout` and silently stops calling (eventually removes) the hook. The sync-over-async is also a latent deadlock: it's safe only because `WindowAnimator.CreateImmediatePlan` returns zero frames; any plan with frames would `await Task.Delay` posting back to the blocked dispatcher. Nothing detects hook death (see M-W2). **Fix:** do inspection/restore work asynchronously off the hook thread; hook callback should only record events.

### H3. macOS: event tap never re-enabled after timeout/user-input disable
`DragRuntimeController.swift:39-43,77-142`

`.tapDisabledByTimeout`/`.tapDisabledByUserInput` fall into `default: break`; `CGEvent.tapEnable` is called once at start. The callback does synchronous AX IPC on every global mouse-down (8+ `AXUIElementCopyAttributeValue` round-trips), so one slow/hung app can push it past the WindowServer timeout — the OS disables the tap and Pop silently stops responding to all drags until restart. Standard fix (what Rectangle-class apps do): handle those event types in the callback and re-enable the tap.

### H4. macOS: update archive consumed after URLSession deletes it
`UpdateService.swift:330-345` (consumed at `:193-194`, `:510`)

`didFinishDownloadingTo` hops to the main actor via `Task { @MainActor ... }` and resumes the continuation with `location` after the delegate method returns — but URLSession deletes the temp file once the delegate returns. The later `moveItem(at: archiveURL, ...)` races the deletion and typically throws, surfacing as "Update check failed" on every update attempt. **Fix:** move the file to a stable path synchronously inside the delegate callback.

### H5. Linux: unguarded polling loop — one exception permanently and silently kills drag tracking
`src/Pop.App.Linux/Platform/X11/X11PollingDragTracker.cs:74-98`

`PollAsync` has no try/catch; it calls inspectors with `checked` casts (`X11WindowInspector.cs:260`, `X11PropertyReader.cs:47`) and invokes host handlers synchronously. Any throw faults the task unobserved, `_pollingTask` stays non-null so `Start()` refuses to restart — Pop stops snapping for the rest of the session with zero diagnostics.

### H6. Linux: no `XSetErrorHandler` — an async X error can terminate the process
`Platform/X11/X11Native.cs` (verified: `XSetErrorHandler` appears nowhere in `src/`)

Pop opens its own `Display` and queries window IDs held across time; if a target window is destroyed mid-drag, the server returns `BadWindow` and the process-wide Xlib default handler prints and calls `exit()`. Pop currently survives only if Avalonia's X11 backend happens to have installed a process-wide handler — incidental, and absent on non-X11 Avalonia paths.

### H7. Linux/X11: multi-monitor support effectively missing
`Platform/X11/X11WindowInspector.cs:47-49`

`InspectMonitorAt` explicitly discards the point (`_ = screenPoint;`) and returns the whole virtual screen (root geometry + `_NET_WORKAREA`). On any multi-monitor setup, "snap left" fills the left half of the *combined* desktop, and Core's cross-monitor logic can never trigger. No XRandR/Xinerama usage exists. Worth stating in README until fixed.

### H8. Core: `JsonSettingsStore.LoadAsync` only catches `JsonException`
`src/Pop.Core/Services/JsonSettingsStore.cs:19-33`

`IOException`/`UnauthorizedAccessException` (file locked by AV/backup, broken permissions) or a delete between `File.Exists` and `File.OpenRead` escapes and can crash host startup instead of falling back to defaults.

### H9. Core: `DiagnosticsLogFormatter` can throw on the drag hot path
`src/Pop.Core/Services/DiagnosticsLogFormatter.cs:12-16`

`ToDictionary` over keys truncated to 48 chars throws `ArgumentException` on duplicate truncated keys — a crash inside the drag-release handler, exactly where the architecture doc says the process must not fail.

## Medium

### Windows host
- **M-W1 — No single-instance enforcement** (`App.xaml.cs:10-17`): manual start + HKCU Run entry ⇒ two global hooks, two tray icons, competing `MoveWindow`/restore calls.
- **M-W2 — Hook death never detected** (`MouseHookDragTracker.cs:33-54`): no liveness check after OS-initiated unhooking (see H2); `UnhookWindowsHookEx` return ignored; `SetWindowsHookEx` P/Invoke lacks `SetLastError = true` (`NativeMethods.cs:83-84`), so startup failures can't report the Win32 error.
- **M-W3 — `UpdateService.Dispose` races in-flight work** (`Services/UpdateService.cs:70-75,97,142`): disposes `_checkGate`/CTS without awaiting `_backgroundTask`; an in-flight check throws `ObjectDisposedException`, reaching an async-void tray handler (`PopHost.cs:71`) with **no** `DispatcherUnhandledException` anywhere ⇒ process crash if "Check For Updates" is clicked during shutdown. `OnUpdateStateChanged` (`PopHost.cs:407-416`) can also hit a null `Application.Current` after `Run()` returns.
- **M-W4 — No global exception safety net** (`App.xaml.cs`): no `DispatcherUnhandledException` / `AppDomain.UnhandledException` / `TaskScheduler.UnobservedTaskException` for a tray-only app.

### Linux host
- **M-L1 — `_snapRestoreStates` dictionary raced across threads** (`LinuxPopHost.cs:183,188-206`): async-void completion writes after an awaited glide while the 8 ms poll loop reads/removes — undefined behavior on a plain `Dictionary` (snap a window, immediately grab another during the glide).
- **M-L2 — Save can crash the app on Plasma** (`SettingsWindow.cs:247-275`, `KWinWaylandIntegration.cs:45,119`): async-void click handler has no catch; `loadScript` failure (`allowFailure: false`, e.g. KWin restarting) escapes onto Avalonia's dispatcher and terminates the process. Also `_settings` is assigned before persisting (`LinuxPopHost.cs:79`), so memory/disk diverge on failure.
- **M-L3 — Unconditional 125 Hz blocking `XQueryPointer` polling even when snapping is disabled** (`X11PollingDragTracker.cs:11,96`; host checks `Enabled` only in handlers): constant CPU/battery cost.
- **M-L4 — Shutdown use-after-free** (`LinuxPopHost.cs:88-104`, `X11WindowMover.cs:37-51`): `Dispose` closes the display while an in-flight glide can still call `XSendEvent` on the freed pointer ⇒ SIGSEGV on quit-during-glide. `X11DisplayConnection.Dispose` is not idempotent.
- **M-L5 — Missing `gdbus` binary throws even on "allowed to fail" calls** (`KWinWaylandIntegration.cs:90-114`): `allowFailure` only applies after exit; `Process.Start` failure makes the whole Plasma path unusable with a poor error, and feeds M-L2 from Save.
- **M-L6 — Linux settings UI silently resets `LaunchAtStartup`** (`SettingsWindow.cs:279-301`): `TryBuildSettings` builds a fresh `AppSettings` with only the five UI fields; any non-UI contract field is wiped on save. (Note `LaunchAtStartup` is also a silent no-op on Linux — no `IStartupRegistration` implementation exists there.)

### macOS host
- **M-M1 — No exception barrier in Native AOT exports** (`MacBridgeExports.cs:7-96`): a managed throw out of an `[UnmanagedCallersOnly]` export is a fail-fast process abort (e.g. garbage timestamp → `DateTimeOffset.FromUnixTimeMilliseconds` throws → menu-bar app dies mid-drag). Wrap each export in try/catch returning safe defaults.
- **M-M2 — Accessibility revocation never detected at runtime** (`DragRuntimeController.swift:263-270`, `AppDelegate.swift:106-111`): revoke AX while running ⇒ tap dies but menu still shows "Accessibility: Granted"; no re-check or recovery until manual toggle.

### Core / abstractions
- **M-C1 — Settings values never validated/clamped after deserialize** (`JsonSettingsStore.cs:27`, `AppSettings.cs:19-26`): `"HorizontalDominanceRatio": 0` defeats the dominance gate (`SnapDecider.cs:101`); negative `GlideDurationMs` flows into a negative-duration animation plan (`WindowAnimator.cs:30`).
- **M-C2 — Ctrl path gates on dominant-axis velocity** (`SnapDecider.cs:118-136`): a fast *vertical* Ctrl-flick with no monitor above/below qualifies as a horizontal half-snap chosen by landing X.
- **M-C3 — No concurrency contract on `ISettingsStore`** (`JsonSettingsStore.cs:36-64`): concurrent `SaveAsync` can throw `IOException` from `File.Replace` on Windows; last-writer-wins otherwise.
- **M-C4 — Corrupt settings silently discarded** (`JsonSettingsStore.cs:30-33`): no diagnostic, no backup; `EnableDiagnostics` resets to false so the evidence is unloggable, then overwritten.
- **M-C5 — `DragSession` is mutable shared state with no thread-affinity contract** (`Models/DragSession.cs:8,40-49`, `IDragTracker`): hook thread `AddSample` vs consumer iterating `Samples` ⇒ collection-modified or torn velocity window; interfaces don't say which thread events fire on.
- **M-C6 — `IStartupRegistration.SetLaunchAtStartup` is void with no error contract or read-back** (`Startup/IStartupRegistration.cs:5`): settings flag can permanently desync from actual OS registration.

### CI / release pipeline
- **M-P1 — No code signing anywhere**: `vpk pack` output unsigned (`release.yml:204-219`); macOS ad-hoc signed only, never notarized (`scripts/build-mac-app.sh:19,35-37`; `POP_MAC_CODESIGN_IDENTITY` never set in CI). Update trust rests entirely on GitHub TLS + write access to the `update-feed` branch.
- **M-P2 — appimagetool downloaded from a moving `continuous` tag, unverified, and executed in a job with a persisted `contents: write` token** (`scripts/package-linux-release.sh:98-105`; `persist-credentials` defaulted on in all workflows): supply-chain path to the auto-update feed. Pin a release + verify a checksum; set `persist-credentials: false` where the token isn't needed.
- **M-P3 — Linux packaging runs after the release is public** (`release-linux.yml:4-8`, `release.yml:300`): a Linux failure leaves a published release with no Linux assets; the Linux `dotnet test` runs too late to gate anything.
- **M-P4 — Asset-cleanup deletes Linux assets on re-runs** (`release.yml:336-343,446-465`): re-running for an existing tag removes previously published `.AppImage`/`.tar.gz`, restored only if `release-linux` later succeeds.
- **M-P5 — No `permissions:` block in `ci.yml`** (should be `contents: read`); **no `concurrency` group in `release.yml`** (two quick bumps race on the `update-feed` push and asset create/delete).
- **M-P6 — No CI leg exercises Linux-specific code** (`ci.yml:14-27`): ubuntu job runs only `tests/Pop.Tests`; `Pop.App.Linux` compiles only incidentally on the Windows runner; no Linux test project exists.
- **M-P7 — AppRun exit-code bug** (`scripts/package-linux-release.sh:76-82`): in the non-terminal failure path, `exit_code=$?` captures the completed `if` statement's status (0), so the AppImage reports failure but exits 0 (verified experimentally).

## Low (abbreviated)

- Core: negative duration not clamped in one `WindowAnimator` branch (`WindowAnimator.cs:30`); 100 px min-size clamp causes visible pop for small windows (`:55-56`); `DiagnosticsLogFormatter` drops fields nondeterministically past 16 and can split surrogate pairs (`:13,27-35`); `ResetDragOrigin` leaves stale release state (`DragSession.cs:80-87`); `plan = default` yields null members if the bool is ignored (`QualifiedSnapPlanner.cs:26`).
- Windows: `_snapRestoreStates` unbounded and keyed by recyclable HWNDs (`PopHost.cs:30,230,235`) — handle reuse can "restore" another app's old bounds; diagnostics log never rotates and logs every ineligible click system-wide (`DiagnosticsLogService.cs:33`, `PopHost.cs:386-397`); `GetWindowLongPtrW` hard-coded entry points break x86 (mitigated: win-x64 RID) and `OverlayNativeMethods.cs` appears unused; Velopack `SimpleWebSource` third ctor arg `3d` overrides the default timeout parameter 10× downward — confirm intent (`VelopackUpdateClient.cs:11`).
- macOS: `"AXResizable"` is not a real AX attribute so every window counts as resizable (`WindowSystem.swift:36`; use `AXUIElementIsAttributeSettable(kAXSizeAttribute)`); `CFHash(window)` restore keys can collide and never evict (`DragRuntimeController.swift:192,331,365-367`); `runProcess` drains the pipe only after `waitUntilExit` — >64 KB output deadlocks the updater (`UpdateService.swift:562-576`); `GetDistanceSquared` multiplies in 32-bit before widening (`MacBridgeRuntime.cs:172-186`); launch-agent plist written but never registered via `SMAppService`/`launchctl` (`LaunchAgentManager.swift:28-44`); the C event-tap callback synchronously calls a `@MainActor` method (`DragRuntimeController.swift:39-43`) — this *does* compile (CI's `swift test` builds `PopMacApp` and is green) and is dynamically safe because the tap source runs on the main run loop, but `MainActor.assumeIsolated` would make the contract explicit and future-proof against stricter checking.
- Linux: KWin `start` also starts unrelated stopped scripts (`KWinWaylandIntegration.cs:52-55`); `Dispose` blocks the UI thread on gdbus without timeout (`:58-71`); 96-byte `XClientMessageEvent` passed where Xlib expects a 192-byte `XEvent` union — benign over-read, should be padded (`X11WindowMover.cs:63-84`); caption heuristic false-positives on CSD apps (`X11WindowInspector.cs:291-304`); settings parse is current-culture — "1.5" parses as 15 on comma-decimal locales (`SettingsWindow.cs:281,287`).
- Repo/CI: actions pinned by major tag only; manual `workflow_dispatch` without `force_release` almost always fails on the version gate (`release.yml:41-63`); four version properties bumped by hand, only `Version` validated (`Directory.Build.props:3-6`); 1.3 MB `official_icon.png` in repo root embedded wholesale as an AvaloniaResource and installed as a "256x256" icon (`Pop.App.Linux.csproj:20`, `package-linux-release.sh:50`) — `artifacts/branding/pop-badge-512.png` (43 KB) is a better fit; shipped `README-linux.txt` contradicts the actual app (says no tray/settings; `Terminal=true` for a tray app) (`package-linux-release.sh:33-43,93`); README says CI runs "on every push/PR" but push trigger is main-only (`ci.yml:4-6`); `package-release.ps1` leaves the GitHub token in a temp clone's `.git/config` (`:138-145`); Swift contract test hardcodes keys instead of reading the contract file (`AppSettingsContractTests.swift:10-32`).

## Test coverage gaps (Pop.Core)

Happy paths are well covered; the edges are not:

- **SnapDecider**: no tests for `InsufficientSamples`, `InvalidSampleWindow` (identical timestamps), the 120 ms velocity-window trimming, the last-4-samples fallback, Ctrl same-monitor slow rejection, the Ctrl vertical-throw case (M-C2), or exact threshold boundaries.
- **SnapRestoreCalculator**: `KeepReachable` clamping entirely untested (off-screen restores, negative-coordinate monitors, degenerate work areas).
- **WindowAnimator**: `durationMs <= 16`, empty bounds, `CreateImmediatePlan`, overshoot clamping, and the 100 px clamp untested.
- **JsonSettingsStore**: no tests for unreadable files, concurrent access, unknown keys, literal `null`, wrong-typed values.
- **WindowEligibilityEvaluator**: only 3 of 10 rejection reasons covered.
- **QualifiedSnapPlanner** cross-monitor path only exercised indirectly via Mac bridge tests; `TileLayoutCalculator` untested with negative-origin monitors.
- Platform: no Linux test project at all; the macOS *app* target (where C1/H3/H4 live) has no tests — only `PopMacSupport` does.

## Verified sound (worth knowing)

- **Windows update flow**: fully delegated to Velopack; feed URL HTTPS and compile-time constant; packages hash-verified against the manifest; app never launches downloaded files itself; `asInvoker`, per-user install (no privilege boundary crossed by local tampering); pending-restart state well-tested.
- **macOS bridge boundary**: all nine struct layouts match C#/C-header/Swift exactly (offsets computed); alloc/free pairs balanced and allocator-matched; no use-after-free on the bridge.
- **Windows hook chain**: callback wraps work in try/catch and always calls `CallNextHookEx`; delegate held in a readonly field (no GC of the hook proc); elevated-window check fails closed.
- **KWin script injection**: only strongly-typed values interpolated invariantly — no injection vector; gdbus invoked via `ArgumentList` (no shell), pipes drained before wait, no zombies.
- **Contract drift**: none — `contracts/app-settings.contract.json` matches `AppSettings.cs` and `AppSettings.swift` exactly, enforced by a file-reading C# test.
- **CI security**: no `pull_request_target`, no untrusted-input interpolation into `run:` blocks, no committed secrets.

## Prioritized recommendations

1. **Fix C1** (mac coordinate spaces) and smoke-test the mac gesture path on hardware; fix H3 (tap re-enable) and H4 (update temp file) in the same pass.
2. **Windows input integrity**: H1 (DPI manifest — one-line fix), H2 (move blocking work off the hook thread), then M-W1/M-W2/M-W4 (single instance, hook liveness, global exception handlers).
3. **Linux resilience**: H5 (guard the poll loop), H6 (X error handler), M-L1/M-L2/M-L4; document the single-monitor limitation (H7) until XRandR support lands.
4. **Core hardening**: H8/H9 (catch broadly in settings load and diagnostics format), M-C1 (clamp settings on load), M-C2 (Ctrl vertical-flick gate).
5. **Release pipeline**: M-P2 (pin + checksum appimagetool, drop persisted credentials), M-P3/M-P4 (build Linux before publishing, stop deleting its assets), M-P5 (permissions/concurrency), and longer-term M-P1 (signing/notarization).
